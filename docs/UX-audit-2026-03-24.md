# UX Audit — EventStormingTool
**Date**: 2026-03-24
**Auditor**: UI/UX Agent
**Scope**: Full application audit post-Bundle-to-DomainEvent-centric migration (v8→v10)

---

## Executive Summary

The tool has a solid visual foundation and a clearly opinionated DDD color language. The recent migration from 4-in-1 Bundle cards to individual StickyNotes with group semantics has introduced the most significant UX gaps. The core problems cluster around three themes: **group identity is invisible** (users cannot perceive that Entity+Command+Information+DomainEvent belong together), **critical actions are hover-only with no fallback** (users who miss the action bar have no other entry point), and **terminology is inconsistent** across surfaces (the panel says "Entity (Aggregate)" while the modal says "Set Entity" and the board shows `◆ OrderPlaced ✓`).

## Audit Findings

---

### [SEVERITY: High] Group membership has no persistent visual indicator

- **Where**: `BoardCanvas.tsx` — `computeGroupBoundingBoxes`, rendered group background div
- **Problem**: The group boundary is a faint dashed rectangle (`rgba(0,0,0,0.03)` fill, `rgba(0,0,0,0.15)` 1.5px dashed border). On the light `#f8fafc` canvas this is nearly invisible, especially at zoom levels below 1.0 and when notes overflow the bounding box during intermediate drag states. A new user has no visual cue that the four note types form a logical unit. There is no label or header on the group container that names the aggregate or event it belongs to. The only hint is spatial proximity.
- **Suggestion**: Increase group background opacity to `rgba(99,102,241,0.06)` (indigo tint) and border to `rgba(99,102,241,0.3)` at 1.5px solid. Add a small header label above the group container (positioned at `top: -18px, left: GROUP_BG_PADDING`) showing the DomainEvent name in 10px muted text. This header should be `pointer-events: none`. Consider a left-border accent (`border-left: 3px solid rgba(99,102,241,0.4)`) as a secondary signal for colorblind users since it conveys grouping without relying on fill color.
- **Effort**: S

### [SEVERITY: High] "Add Command" and "Set Entity" actions are hover-only with no keyboard path

- **Where**: `StickyNote.tsx` — `domainEventActionBar`, rendered `bottom: -32` on hover/select
- **Problem**: The action bar appears below a DomainEvent note only on `isHovered || isSelected`. There is no way to trigger these actions from the keyboard. The buttons have no `tabIndex`, no `aria-label`, and no focus state. If a user never hovers the note (touch screen simulation, keyboard-only, or reduced-motion assistive technology context), they will never discover that "+ Command" and "Set Entity" exist. There is also no entry point for these actions in the DetailPanel when no command has been linked yet — the panel only shows an italic gray placeholder: "No command linked — use [+ Command] on the note." This is a dead end for users who read the panel first.
- **Suggestion**: Add "+ Add Command" and "+ Set Entity" action buttons directly in the `DomainEventPanel` inside DetailPanel.tsx, positioned under the "Linked Command" and "Linked Entity" sections respectively when those slots are empty. This gives keyboard users, panel-first users, and confused users a clear secondary entry point. On the note action bar, add `tabIndex={0}` and `onKeyDown` (Enter/Space to activate) to both buttons. Add `role="toolbar"` to the action bar container.
- **Effort**: M

### [SEVERITY: High] No empty-board guidance for a blank new context

- **Where**: `BoardCanvas.tsx` — the `isEmptyState` condition only fires when `isPathFilterActive && filteredCount === 0`. A completely empty board (no notes at all, no active path filter) shows nothing.
- **Problem**: When a user creates a new Context from the Homepage or via the "+" tab button, they land on a completely blank canvas with no indication of what to do. There is no onboarding prompt, no suggested first step, and no arrow pointing to the sidebar palette. The HintBar shows three shortcuts (`F`, `Esc`, `Del`) but none of them help a first-time user understand the DomainEvent → Command → Entity flow. The sidebar palette lists elements without any ordering explanation.
- **Suggestion**: Add an empty-board overlay (similar to the existing path-filter empty state) that renders when `activeBoard.notes.length === 0 && activeBoard.remodels.length === 0 && !isPathFilterActive`. The overlay should show:
  1. A brief prompt: "Start by placing a Domain Event"
  2. A color legend row showing orange (DomainEvent) → blue (Command) → yellow (Entity) with arrows
  3. A single CTA button "Place Domain Event" that activates the DomainEvent tool in the sidebar (calls `setActiveToolType('DomainEvent')`) and dismisses the overlay
  The overlay should be `pointerEvents: none` except for the CTA button.
- **Effort**: M

### [SEVERITY: High] "Set Entity" modal always creates a new Entity note — cannot select an existing one

- **Where**: `SetEntityModal.tsx`, `boardStore.ts` — `addEntityForEvent`
- **Problem**: The SetEntityModal only accepts a free-text name and always creates a brand-new Entity note positioned above the group. There is no way to link an existing Entity or AggregateRoot note on the canvas to a DomainEvent. If a user has already placed an AggregateRoot note or wants two events to share the same entity, they cannot express this relationship through the UI. The `linkEntityToEvent` action exists in the store but is never called from any UI surface.
- **Suggestion**: Extend the SetEntityModal to show a dropdown of existing Entity/AggregateRoot notes on the current board (derived from `allNotes.filter(n => n.type === 'Entity' || n.type === 'AggregateRoot')`). Add a tab switcher: "New Entity" (current behavior) / "Link Existing" (shows dropdown). When "Link Existing" is selected and confirmed, call `linkEntityToEvent(eventId, selectedEntityId)` instead of `addEntityForEvent`. This removes a significant capability gap.
- **Effort**: M

### [SEVERITY: Medium] Terminology is inconsistent across surfaces for the same concept

- **Where**: Multiple components
- **Problem**: The same concept is referred to by four different names:
  - DetailPanel: `SectionLabel` reads "Linked Entity (Aggregate)"
  - DetailPanel: empty state text says "No entity linked — use [Set Entity] on the note"
  - Action bar button on DomainEvent: shows "Set Entity" (before) and `◆ [entityLabel] ✓` (after)
  - StickyNote rendering: when an Entity has a linked AggregateRoot, the type label changes to "AggregateRoot" and shows "via [entityLabel]" underneath — which is the reverse of what a user would expect
  - elementTypes.ts: both `Entity` and `AggregateRoot` exist as separate types, both colored `#FFD600`
  - HintBar: contains Chinese text ("關閉面板", "刪除") mixed with English shortcuts — this is a localization inconsistency, not intentional bilingual design
- **Suggestion**: Standardize on "Entity" across all surfaces for the top-yellow slot. Reserve "AggregateRoot" for the standalone palette type that can be independently placed. The section label in DetailPanel should read "Linked Entity". The empty state hint should say "No entity linked. Use [Set Entity] below." (pointing to the panel action rather than requiring canvas interaction). The HintBar labels should be English consistently: "Close Panel", "Delete".
- **Effort**: S

### [SEVERITY: Medium] "Add Command" is blocked after a command already exists — no edit path

- **Where**: `StickyNote.tsx` — `domainEventActionBar`: `{!linkedCommandNote && (<button>+ Command</button>)}`
- **Problem**: Once a DomainEvent has a linked Command, the "+ Command" button disappears entirely. There is no way to:
  1. Edit the Command's label after creation (the Command note itself can be double-clicked on canvas, but this is not discoverable)
  2. Unlink or replace the Command (no "Remove Command" affordance)
  3. Know that you can double-click the Command note on canvas (there is no tooltip or hint)
  The DetailPanel shows the linked command name in a blue chip but provides no edit button.
- **Suggestion**: When a command is linked, replace the "+ Command" button in the action bar with a pencil/edit icon button (or a chip showing the command name). Clicking it should either: (a) open an edit modal pre-filled with the current command label and information, or (b) scroll the canvas to the Command note and select it. For (a), the simplest path is to add an optional `editMode` prop to AddCommandModal that pre-populates the fields. In the DetailPanel, add a small "Edit" button on the linked command chip.
- **Effort**: M

### [SEVERITY: Medium] Group drag-start has no visual confirmation that the whole group moves

- **Where**: `Board.tsx` — `handleDragStart`, `StickyNote.tsx` DragOverlay
- **Problem**: When a user starts dragging a DomainEvent note (which triggers group movement), the DragOverlay only shows a single note ghost — not the full group. The other group members (Command, Information, Entity) snap to their new positions only on `dragEnd`. During the drag, there is no visual indication that the entire group will move together. This is particularly confusing because:
  1. Dragging an individual Command note also moves the group (via `groupEventId` logic)
  2. The dashed group boundary stays in place during drag
  3. There is no cursor change or tooltip indicating group behavior
- **Suggestion**: On `dragStart`, apply a visual highlight (e.g., a brighter group boundary `border: 2px solid rgba(99,102,241,0.6)` and a slightly elevated `box-shadow` on all group members) to signal that the group moves as a unit. This can be achieved by passing `isDraggingGroupEventId` from Board state down to BoardCanvas and StickyNote. The DragOverlay showing only one note is a known dnd-kit limitation; compensate with the group highlight approach.
- **Effort**: M

### [SEVERITY: Medium] Delete button (×) on notes is only 20×20px — below minimum touch target

- **Where**: `StickyNote.tsx` — `actionBtnStyle`: `width: 20, height: 20`
- **Problem**: The delete button rendered at the top-right corner of selected notes is 20×20px. The WCAG 2.5.5 recommended minimum is 44×44px for touch targets. Even on desktop, a 20px target at the corner of a note that appears only when selected is easy to miss and easy to accidentally trigger. There is no undo for note deletion. Deleting a DomainEvent cascades to delete its entire group — Command, Information, Entity — without any confirmation or warning.
- **Suggestion**: Two changes needed:
  1. Increase the delete button's visual and hit area to at minimum 32×32px using `padding` to expand the tap target while keeping the icon small. The button can remain 20×20px visually but should have an invisible hit area expanded via padding.
  2. Add a confirmation step for DomainEvent deletion that warns "Deleting this event will also delete its linked Command, Information, and Entity notes." Show a `window.confirm()` or an inline confirmation chip before cascading delete. This prevents accidental group destruction.
- **Effort**: S

### [SEVERITY: Medium] Information note body is read-only on canvas — double-click does nothing useful

- **Where**: `StickyNote.tsx` — `isInformation` rendering branch (lines 514–547)
- **Problem**: Information notes display a property list derived from `note.information`. The note renders a non-editable list of `attrName: type` pairs. Double-clicking opens a textarea that edits `note.label`, which is not what's displayed — the displayed content is `note.information[]`. The textarea editing path (`setIsEditing(true)`) results in a confusing experience where the user types in a field that doesn't correspond to what they see.
  Additionally, the FormatToolbar is suppressed for Information notes (`showFormatToolbar` excludes `isInformation`), but the note still supports double-click editing, creating an inconsistent experience.
- **Suggestion**: For Information notes, double-click should open the DetailPanel to the Information section (call `onDetailClick`) rather than entering textarea edit mode. Alternatively, disable double-click entirely on Information notes and show a tooltip on hover: "Edit properties in the detail panel →". This makes the DetailPanel the single source of truth for structured data.
- **Effort**: S

### [SEVERITY: Medium] "ReadModel" exists in the palette as a StickyNote type, but the sidebar removes it in favor of "Remodel"

- **Where**: `SidebarPalette.tsx` line 341: `ELEMENT_TYPE_LIST.filter((t) => t !== 'ReadModel')`
- **Problem**: `ReadModel` is in `ELEMENT_TYPE_LIST` but filtered out of the sidebar. The comment says it is "replaced by Remodel 4-in-1". However:
  1. ReadModel notes can still be created via MCP (AI agent), resulting in notes that appear on the canvas but have no corresponding palette entry and no special rendering logic
  2. The Remodel component is conceptually different from a ReadModel note — Remodel is a 4-panel query card while ReadModel was a simple note. Silently replacing one with the other is confusing for users who know DDD terminology
  3. There is no explanation in the sidebar why there is no "Read Model" note type — a new user might search for it
- **Suggestion**: Either: (a) keep a grayed-out "Read Model" entry in the palette with a tooltip "Use the Read Model card (⊟) instead", or (b) rename the "Read Model" Remodel button to "Read Model Card" and add a subtitle "(Query / Return type)" to clarify it is not a simple note. Option (b) is cleaner.
- **Effort**: S

### [SEVERITY: Medium] AddCommandModal confirms with Cmd+Enter but SetEntityModal confirms with Enter — inconsistent keyboard shortcuts

- **Where**: `AddCommandModal.tsx` line 74, `SetEntityModal.tsx` line 55
- **Problem**:
  - `AddCommandModal`: Enter key in the command name input does NOT submit. Only `Cmd/Ctrl+Enter` submits.
  - `SetEntityModal`: Enter key in the entity name input DOES submit directly.
  The two modals have different confirmation behaviors for essentially the same interaction pattern. This inconsistency will cause users to make mistakes when switching between the two modals.
- **Suggestion**: Standardize both modals to confirm on `Enter` when the primary text input is focused, and `Escape` to cancel. The reason AddCommandModal deviates is the property table where Enter might add rows, but the command name field itself should be Enter-to-confirm. The property table rows should use Tab to move between fields and a dedicated "+ Add Property" button remains the row-add action.
- **Effort**: S

### [SEVERITY: Medium] No save/sync status indicator

- **Where**: Global — no status indicator component exists
- **Problem**: The application auto-saves to localStorage on every state change and syncs to the MCP server via POST `/api/board`. Users have no visibility into whether their changes have been saved. If the POST request to the Express server fails (server not running, network issue), the failure is silent. There is no "Saved" confirmation, no error toast, and no "Unsaved changes" warning.
- **Suggestion**: Add a minimal status indicator in the TabBar (right side, before the statistics chip). States:
  - Default (no pending changes): invisible or a faint "Saved" text
  - Syncing: small animated spinner + "Saving..."
  - Success: brief "Saved" flash that fades after 1.5s
  - Error: red "Sync failed" with a retry button
  This leverages the existing `apiSync.ts` SSE/POST mechanism — hook into the POST response to drive the indicator state.
- **Effort**: M

### [SEVERITY: Low] Aggregate vs AggregateRoot — two palette items that both produce yellow notes

- **Where**: `ELEMENT_CONFIGS` — `Aggregate` (color: `#FFD600`) and `AggregateRoot` (color: `#FFD600`); `ELEMENT_TYPE_LIST` includes both; sidebar palette includes `AggregateRoot` but filters out `ReadModel`
- **Problem**: `Aggregate` and `AggregateRoot` share the exact same color, label prefix, and default size. The distinction between them is not communicated in the UI. Users who place an `Aggregate` note will be confused because it behaves differently from an `Entity` that has been "Set" via the DomainEvent action bar — the former is a standalone note, the latter is a group member that gets `groupEventId`. The migration code (v9→v10) even promotes `Aggregate` to `Entity` when linked to a DomainEvent's `entityId`, showing that the semantic boundary is blurry.
- **Suggestion**: Clarify in the sidebar with a sub-label under `AggregateRoot`: "(standalone — not linked to event)". Give `Aggregate` a slightly different shade (e.g., `#FFC107` amber vs `#FFD600` yellow) if both must coexist to visually distinguish them. Longer term, consolidate — the model no longer needs both `Aggregate` and `AggregateRoot` as separate types post-migration.
- **Effort**: S–M (visual only = S; consolidation = M)

### [SEVERITY: Low] Resize handle is invisible until hover — affordance is only the cursor change

- **Where**: `StickyNote.tsx` — `resizeHandle`, renders an 8×8 SVG path corner indicator
- **Problem**: The resize handle is an 8×8 SVG at the bottom-right corner of every note. The SVG lines are `rgba(0,0,0,0.3)` — nearly invisible on colored note backgrounds (dark orange DomainEvent, dark blue Command). The only resize affordance for users is the cursor change to `se-resize`, which requires hovering over exactly the right corner. New users will not discover this feature.
- **Suggestion**: Make the resize handle always visible (not opacity: 0 until hover) at `rgba(0,0,0,0.2)` and increase it to 12×12px. On hover of the note card, increase handle opacity to `rgba(0,0,0,0.5)`. This follows standard whiteboard tool conventions (Miro, Figma all show corner handles on hover of the note, not just at the exact corner).
- **Effort**: S

### [SEVERITY: Low] Path assignment requires opening DetailPanel — no canvas-level shortcut

- **Where**: `DomainEventPanel` in `DetailPanel.tsx` — path assignment via checkboxes; no path assignment on other note types' panels
- **Problem**: To assign a note to a flow path, users must: click the note → wait for DetailPanel to open → scroll to the "Paths" section → check the path checkbox. There is no right-click context menu, no drag-to-path affordance, and no batch assignment tool. For a session with 20+ DomainEvents, assigning paths one by one is extremely tedious.
- **Suggestion**: (Phase 2 item, but worth noting now): Add a right-click context menu on any note with "Assign to Path..." that shows a quick picker of existing paths. Also consider a "bulk assign" mode that lets users Shift-click multiple notes and assign them all to a path via the DetailPanel. For MVP: ensure the path section is visible at the top of the DomainEvent panel (currently scrolled below the properties section) so users discover it faster.
- **Effort**: M (context menu) / S (reorder panel sections)

- **Problem**: Based on the component being rendered at the bottom-right of the Board, there is no visible label or tooltip explaining that the minimap can be clicked/dragged to navigate. Users who have not used whiteboard tools before will not know it is interactive.

### [SEVERITY: Low] Homepage delete is a double-click "click to confirm" pattern with no time feedback

- **Where**: `Homepage.tsx` — `handleDelete` two-click confirm pattern
- **Problem**: Clicking the trash icon once turns it red and shows "Confirm". Clicking elsewhere resets it (via the outer `onClick={() => setConfirmDeleteId(null)}`). This pattern is common but has no time limit — the red state persists indefinitely until the user clicks away. More importantly, there is no explanation of what will be deleted (the board and all its notes, including actor sub-boards). The button tooltip says "Click again to confirm delete" but only appears on hover, not in the confirm state.
- **Suggestion**: When in confirm state, add a small text below the card: "This will permanently delete all notes in this context." This prevents accidental destruction. Also consider adding a 5s auto-reset timer so the red state does not persist indefinitely.
- **Effort**: S

## Accessibility Findings

### Color-only meaning for note types
Every note type is identified primarily by background color (orange = DomainEvent, blue = Command, yellow = Entity, etc.). Users with deuteranopia (red-green colorblindness) may confuse `#FFD600` (Entity/Aggregate, yellow) with `#FF8C00` (DomainEvent, orange) — these fall close together on the tritanopic spectrum. The type label in the top-left corner of each note (`fontSize: 10px, opacity: 0.8`) is the only non-color signal, but it disappears for `Information` notes and is visually weak at small zoom levels.

**Suggestion**: Ensure type label font size is never below 8px at any zoom and consider adding a distinct icon (not emoji) per type as a secondary signal in the note header area.

### Touch target violations
- Delete button: 20×20px (minimum 44×44px recommended)
- Path tab close button: not audited but likely similar
- Action bar buttons ("+ Command", "Set Entity"): 20px height with `padding: '3px 8px'` = approximately 26px total height — below threshold

### Keyboard navigation
- No `tabIndex` on the canvas drag-and-drop area
- No keyboard shortcut to enter "place note" mode (selecting DomainEvent tool) without clicking the sidebar
- No `aria-label` on icon-only buttons in the TabBar (home icon `⌂`, add board `+`, close tab `×`)
- FormatToolbar has no documented keyboard path

## Flow Completeness Audit

### Flow: DomainEvent → Command → Entity (first-time user)
| Step | Current behavior | Gap |
|------|-----------------|-----|
| 1. Place DomainEvent | Click palette, click canvas | No guidance that this is step 1 |
| 2. Add Command | Hover DomainEvent, click "+ Command" | Hover-only, not discoverable |
| 3. Name Command | Modal appears, Enter does not confirm | Enter key inconsistency (see finding above) |
| 4. Set Entity | Hover DomainEvent, click "Set Entity" | Hover-only, creates new Entity only |
| 5. Link events | Activate Link mode in sidebar, click two events | Link mode has no visual tutorial |
| 6. Assign to path | Click note, open detail panel, scroll to Paths | Multi-step, no canvas shortcut |

**Dead ends identified**:
- User places an `AggregateRoot` note and wants to link it to a DomainEvent → no UI path for this (linkEntityToEvent never called from UI)
- User wants to edit a Command's label after creation → must find and double-click the Command note on canvas
- User wants to delete just the Entity from a group → deletes the Entity note, but the DomainEvent still has `entityId` pointing to a now-deleted note (the store clears `entityId` on cascaded DomainEvent delete but not on direct Entity delete)

**Note on the last dead end**: In `deleteNote`, the cleanup loop (`for (const note of board.notes)`) clears `commandId` and `entityId` from DomainEvent notes when the referenced note is deleted. This correctly handles Entity note deletion (the `entityId` on the DomainEvent will be cleared). This is correctly implemented. I am retracting the dead-end claim for this specific case.

### Flow: Command already linked — edit or replace
- User clicks "+ Command" → it is absent (hidden when `linkedCommandNote` exists) — **dead end confirmed**
- Only workaround: delete the DomainEvent (losing everything), or double-click the Command note

## Patterns Done Well — Preserve These

1. **DomainEvent cascade delete**: When a DomainEvent is deleted, `boardStore.deleteNote` correctly cascades to remove all `groupEventId`-linked notes and clears `commandId`/`entityId` backreferences. This is architecturally sound and prevents orphaned notes.

2. **Path filter dimming**: When a path filter is active, non-path notes get `opacity: 0.15`, `saturate(0.3)`, and `scale(0.97)` with `transition: 200ms ease`. This is a polished, non-destructive focus mode that preserves spatial context.

3. **Modal form quality**: Both `AddCommandModal` and `SetEntityModal` auto-focus the primary input, reset on open, have disabled states with appropriate visual feedback, and support `Escape` to close. The dark modal theme on a dark overlay (`rgba(0,0,0,0.7)`) is appropriately high-contrast.

4. **DetailPanel DomainEvent view**: The tab switcher between "Command Input" and "Event Output" is a thoughtful information architecture choice that prevents overwhelming the user. The "Linked Command" chip provides clear relationship tracing.

5. **Group bounding box computation**: `computeGroupBoundingBoxes` in `BoardCanvas.tsx` correctly includes the DomainEvent anchor in the bounding box (not just satellites) and adds padding. The logic is correct and efficient.

6. **Homepage health dots**: The `getHealth` signal (green/yellow/red) in the sidebar and homepage gives a quick at-a-glance project status. The three states (empty, has notes but no links, complete) map well to Event Storming session maturity.

7. **Fit All shortcut (F key)**: The `F` key shortcut for fit-all is context-aware (ignores key when input is focused) and re-calculates based on actual note bounds. The implementation in `BoardCanvas.tsx` is correct.

## Top 5 Most Impactful Improvements

Ranked by impact × feasibility, these five items will meaningfully improve the tool for first-time users and regular users alike.

### Priority 1 — Empty board onboarding overlay
**Impact**: Eliminates the blank canvas confusion that affects 100% of first-time users.
**Why now**: Requires only a new conditional render in `BoardCanvas.tsx` with no state changes or architectural impact. One engineer, one afternoon.
**Spec pointer**: See finding "No empty-board guidance" above.

### Priority 2 — Add Command / Set Entity buttons in DetailPanel (not just action bar)
**Impact**: Removes the most critical dead end (hover-only discovery) and unblocks keyboard users.
**Why now**: Both `addCommandForEvent` and `addEntityForEvent` already fire from `Board.tsx`. The DetailPanel just needs buttons wired to call `setAddCommandForEventId`/`setSetEntityForEventId` — the modal machinery already exists.
**Spec pointer**: See finding "Add Command actions are hover-only" above.

### Priority 3 — Group visual identity (border opacity + event name header)
**Impact**: Directly solves the primary conceptual gap introduced by the Bundle-to-StickyNote migration: users do not know that notes form a group.
**Why now**: CSS-only change to the group container in `BoardCanvas.tsx` + a small positioned label. No state changes needed.
**Spec pointer**: See finding "Group membership has no persistent visual indicator" above.

### Priority 4 — Terminology standardization + HintBar English labels
**Impact**: Reduces cognitive load from inconsistent naming and language mixing. QA-testable string changes.
**Why now**: Pure text changes across DetailPanel, HintBar, and DomainEvent action bar. Zero risk.
**Spec pointer**: See finding "Terminology is inconsistent across surfaces" and "HintBar contains mixed-language labels" above.

### Priority 5 — SetEntityModal: add "Link Existing" tab
**Impact**: Enables a core DDD modeling workflow: multiple events can share the same Aggregate Root. Without this, the model will always create duplicate Entity notes.
**Why now**: The `linkEntityToEvent` store action already exists. The modal needs a dropdown of existing Entity/AggregateRoot notes. Medium effort but high semantic value.
**Spec pointer**: See finding "Set Entity modal always creates a new Entity note" above.
